Request Flow — How a Question Becomes a Query

1

User types a natural language question

e.g. "Show me all invoices with status INVC from last month"
The Next.js frontend sends a POST /api/chat request with the question and a UUID session ID.

2

Load conversation history (Memory Service)

FastAPI reads prior turns for the session from Redis (or falls back to an in-memory dictionary if Redis is unavailable). History is formatted as User: … / Assistant: … pairs and injected into the prompt for context continuity.

3

Retrieve schema context (RAG Service)

The question is embedded using nomic-embed-text via Ollama. Cosine similarity is computed against 2,530 pre-indexed table embeddings stored in vector_store.json. Tables with a name-prefix match to query words receive a score boost. The top 8 most relevant table schemas are returned as context.

4

Generate SQL (LLM Service)

A structured prompt is sent to sqlcoder:7b-q4_0 via Ollama (temperature = 0 for deterministic output). The prompt contains: dialect rules, the top-8 schema context, conversation history, and the user's question. The LLM output is cleaned of markdown fences and prompt artifacts. The model is instructed to only use table/column names from the provided schema.

5

Validate & execute SQL (SQL Service)

The generated SQL is checked against a disallowed-statement regex (INSERT, UPDATE, DELETE, DROP, etc.). If it passes, SQLAlchemy executes it against MSSQL via pyodbc. Results are capped at MAX_RESULT_ROWS=500.

6

Explain the SQL (LLM Service)

A second, lighter LLM call generates a plain-English explanation of what the SQL does and why it answers the question (2–4 sentences).

7

Save turn & return response

The full turn (question + SQL + explanation) is persisted back to Redis/memory. The response — containing sql, explanation, result (columns + rows), and any error — is returned to the frontend as JSON.

8

Render in UI

The frontend renders the SQL query in a code block, the explanation as prose, and the result rows in a sortable data table. Errors are shown inline with the assistant message.

RAG Pipeline — Schema Indexing & Retrieval

🗄️
MSSQL Introspection
Single bulk query reads INFORMATION_SCHEMA for all tables, columns, PKs & FKs in schema dbo.
→
📝
Text Serialisation
Each table is formatted as:
Table: INVC\nColumns:\n - INVOICE (char) [PK]…
→
🔢
Embed (Ollama)
Each table text is sent to nomic-embed-text. Up to 3 retries per text. Truncated to 2000 chars.
→
💾
Persist to JSON
Embeddings saved in batches to chroma_data/vector_store.json. Survives server restarts.
→
🔍
Query-time Retrieval
Question embedded → cosine similarity vs all 2,530 vectors → name-prefix boost → top-8 returned.

Name-prefix boost logic:   If a table's exact name appears in the query → +0.30 score.   If the first 3 chars of the table name appear inside any query word → +0.15 score.   Example: query "invoice details" → "INV" prefix found in "invoice" → INVC table gets +0.15, rising from rank #26 to #1.

Backend Services Detail

🧠
LLM Service
llm_service.py
  • Model: sqlcoder:7b-q4_0 via Ollama
  • Temperature 0 — deterministic SQL output
  • generate_sql() — builds structured prompt, calls LLM, strips output artefacts
  • explain_sql() — generates plain-English explanation
  • LangChain PromptTemplate + OllamaLLM chain
🔍
RAG Service
rag_service.py
  • Embedding model: nomic-embed-text
  • Pure-Python cosine similarity (no ChromaDB/C++)
  • index_schema() — introspects DB, embeds, persists JSON
  • retrieve_schema_context() — top-8 with name-prefix boost
  • JSON vector store survives restarts
💬
Memory Service
memory_service.py
  • Primary: Redis async (redis.asyncio)
  • Fallback: in-memory dict (no Redis needed)
  • TTL: 86,400 s (24 h) per session
  • Session key: nl2sql:session:{uuid}
  • Stores role|content entries in Redis list
🗄️
SQL Service
sql_service.py
  • SQLAlchemy engine with pyodbc (ODBC Driver 17)
  • Connection pool: size 5, overflow 10
  • Safety regex blocks INSERT/UPDATE/DELETE/DROP/EXEC…
  • get_schema_info() — single bulk INFORMATION_SCHEMA query
  • Results capped at MAX_RESULT_ROWS=500

Resilience & Fallbacks

🔴 Redis unavailable

Redis connection fails on startup
→
Falls back to in-memory dict
→
Conversations work normally (reset on restart)

⚠️ Embedding fails during retrieval

Ollama embed call fails
→
Returns first 8 store entries as context
→
Query still attempted with degraded schema context

🛡️ SQL safety guard

LLM generates non-SELECT SQL
→
SQLSafetyError raised
→
Error returned to user, DB never touched

📄 Schema indexing partial failure

Indexing fails mid-way
→
Already-saved batches preserved on disk
→
Partial count reported in API response
